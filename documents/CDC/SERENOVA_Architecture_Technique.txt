1.  VUE D'ENSEMBLE DE L'ARCHITECTURE

?  1.1  Philosophie & Contraintes
L'architecture de SÉRÉNOVA est conçue autour de trois impératifs fondamentaux :

• Distribution sans store : génération d'un APK Android installable directement + accès via navigateur web — aucune dépendance à Google Play ou à l'App Store Apple.
• Codebase unique : le même code source produit l'application Android ET l'application web, minimisant la maintenance.
• Scalabilité progressive : conçue pour 10–30 unités, mais l'architecture ne pose aucun plafond technique — extensible à des centaines d'unités sans refonte.

?  1.2  Décision Architecturale : React Native (Expo) + Supabase
Après analyse des contraintes (APK direct, web app, équipe réduite, démarrage rapide), la stack retenue est :

?  1.3  Pourquoi Supabase plutôt qu'un backend custom ?
• Gratuit pour démarrer (plan Free : 500 Mo DB, 1 Go storage, 50 000 utilisateurs actifs/mois)
• PostgreSQL managé avec Row Level Security (RLS) — isolation des données par utilisateur
• SDK JavaScript natif — intégration directe dans React Native sans backend intermédiaire
• Realtime intégré — tableaux de bord mis à jour instantanément
• Auth prête à l'emploi — JWT, sessions, refresh tokens, social login extensible
• Edge Functions (Deno) — pour PDF, SMS, emails sans serveur dédié
• Scalabilité : upgrade vers plan Pro transparent, sans changement de code

?  2.3  Flux Offline (Mode Hors-Ligne)
SÉRÉNOVA fonctionne partiellement sans connexion internet :

• Consultation des données : les données récemment consultées sont mises en cache par TanStack Query (persistance via AsyncStorage)
• Saisie hors-ligne : les actions (paiement, note) sont stockées dans une file d'attente locale (Expo SQLite)
• Synchronisation automatique : dès la reconnexion, la file est vidée et les données synchronisées avec Supabase
• Indicateur visuel : bandeau "Mode hors-ligne" affiché et badge sur les actions en attente de sync

?  3.3  Sécurité des Données : Row Level Security (RLS)
Supabase impose des règles RLS sur chaque table. Aucune donnée n'est accessible sans authentification valide :

• users : chaque utilisateur ne voit que son propre profil
• sites : SELECT/INSERT/UPDATE/DELETE uniquement sur les sites dont user_id = auth.uid()
• espaces, loyers : accès via la relation sites ? user_id = auth.uid()
• locataires : accès si au moins un bail actif lie le locataire à un espace de l'utilisateur
• paiements, alertes : accès via baux ? espaces ? sites ? user_id = auth.uid()

Aucun accès direct à la base de données n'est possible depuis l'extérieur — toutes les requêtes passent par le SDK Supabase authentifié.

5.  INTERFACES & EDGE FUNCTIONS

?  5.1  Accès aux Données — Supabase SDK
Il n'y a pas de backend REST custom à maintenir. Les données sont accédées directement via le SDK Supabase, avec sécurité RLS. Exemples de patterns utilisés :

OpérationCode SDKLire tous les sitessupabase.from('sites').select('*').eq('user_id', uid)Lire espaces d'un sitesupabase.from('espaces').select('*, loyer(*)').eq('site_id', siteId)Créer un paiementsupabase.from('paiements').insert({ bail_id, montant, ... })Upload photosupabase.storage.from('photos').upload(path, file)Abonnement realtimesupabase.channel('alertes').on('postgres_changes', ...).subscribe()Invoquer Edge Functionsupabase.functions.invoke('gen-quittance', { body: { paiement_id } })
?  5.2  Edge Functions Supabase
Les Edge Functions (Deno) permettent d'exécuter du code serveur sans déployer un backend :

FonctionDéclencheurDescriptiongen-quittancePOST (app)Génère un PDF quittance de loyer à partir d'un paiement_id. Upload dans Storage, retourne l'URL signée.send-relancepg_cron / appEnvoie SMS (Twilio) et/ou email (Resend) de relance pour un bail_id donné. Enregistre l'alerte comme ENVOYEE.gen-rapportPOST (app)Génère le rapport mensuel PDF (récapitulatif encaissements) pour un site et un mois donnés.check-impayespg_cron (02:00)Vérifie chaque nuit les échéances dépassées, crée les alertes IMPAYE manquantes, déclenche send-relance si J+7/15/30.check-bauxpg_cron (03:00)Vérifie les baux dont la date_fin approche (90j, 30j, 15j). Crée les alertes BAIL_EXPIRANT.notify-pushTrigger DBDéclenché par un trigger PostgreSQL sur INSERT dans alertes. Envoie une notification Expo Push au propriétaire.
?  5.3  Planification Automatique (pg_cron)
Les tâches récurrentes sont planifiées directement dans PostgreSQL via l'extension pg_cron (incluse dans Supabase Pro) :

TâcheCronActionVérif. impayés0 2 * * *Chaque nuit à 02h00 — détecte les loyers non payésVérif. baux expirants0 3 * * *Chaque nuit à 03h00 — alerte sur baux proche échéanceRelances J+70 9 * * *Chaque matin à 09h00 — envoi SMS/email pour J+7 impayésRelances J+15/300 10 * * 1Chaque lundi à 10h00 — relances renforcéesNettoyage cache0 4 * * 0Chaque dimanche — purge des alertes RESOLUES > 90 jours
